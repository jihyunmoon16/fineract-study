# 01. Idempotency

## Why This Matters

In a wallet, duplicate requests can create duplicate withdrawals, transfers, or top-ups.
Idempotency is one of the first safety controls to design.

## What To Look For

- where the request identity is stored
- how processed commands are looked up
- whether request and response payloads are persisted
- what state transitions exist for a command

## Questions For My Wallet

- What is my idempotency key?
- Is it generated by client or server?
- How long should it be retained?
- Do I return the original response or an error when the same key is reused?
- What if the first request is still processing?

## Interview Answer Seed

`For money movement APIs, I would require an idempotency key on write requests. The server stores the request identity and processing result. If the same request is retried, I return the previously recorded result instead of executing the transfer again.`

## Failure Scenario

If idempotency is missing:

- client retries on timeout
- server processes both requests
- user gets double debit or duplicate transfer
